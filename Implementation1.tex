\section{SMART goal 1: Setup of an Improved DQM Report}

In Project 1, the groundwork for researching the reporting pipeline was completed. It provided the data lineage and described the underlying issues. Based on this work, an improved report was developed. Research was conducted on how to better highlight the relevant comparisons and help users identify the information more quickly. Additionally, the third-party data source, which was previously copied daily into an Excel file, was improved. The file was moved to SharePoint and is now updated whenever changes are made to \gls{AWSS3}, where it is made available to \gls{athena}. In addition, the selected data underwent a structural change.

\subsection*{Update of data source}

The first step towards the new report was to improve one of the data sources used by the report. Part of the data was retrieved from a third-party system by means of an Excel table. The file resided in a shared folder and was not initially archived. As this data source did not change and was required by both the existing SAC report, which had to remain operational, and the new Power BI report, it had to be made available in a way that was accessible to both tools. Changing from a shared folder, which was not accessible to Power BI, to SharePoint was therefore a major improvement, as it added versioning to the file. Furthermore, the \gls{DWHLSOPS} team has an existing method to transfer .csv data from SharePoint to \gls{athena}.

Therefore, the destination of the Excel file was updated in collaboration with the business users responsible for the manual maintenance of the Excel file. With this change in place, the first data source was made available to Power BI and all versions of it were from now on stored on S3.

\subsubsection*{Initial Data Setup}

As this was the first report developed by the project team, the initial implementation was carried out locally without deploying the report to \gls{microsoftFabric}. As a first report, the variety of available approaches for implementing features such as links between data sources naturally led to a report setup that performed some tasks twice, for example linking the two data sources using both \gls{mQuery} and \gls{dax}. These deficiencies of the initial implementation were cleaned out trough out the further implementation steps of the report. 


\subsubsection*{Differences Between Local and Cloud}

Once the linkage of the two data sources was working and the data was displayed in the same way as in the SAC report, the Power BI report was published to the cloud. However, the report showed issues when loading data from \gls{athena}. Power BI automatically applied a transformation that attempted to cast a column containing date values into a date format. This resulted in issues with empty values.

According to the initial research using \gls{copilot}, the issue could be resolved in Athena. However, discussions with the wider \gls{DWHLSOPS} team showed that similar issues had occurred for other team members. Therefore, before making changes in Athena, the possible solutions identified by the team were investigated. After consulting the team and checking the view in Athena, it became clear that the issue did not originate in Athena and therefore could not be resolved there. As the tests showed that the query ran correctly in Athena, the issue had to be located in the report. Since no proper checkpoints had been established up to this point, it was not possible to determine at which stage the applied data transformations caused the issue. Therefore, the report had to be recreated, and after each transformation, a test deployment was performed to verify that the data loaded and was displayed correctly in the cloud version.

\subsubsection*{New View Athena}
To highlight some of the testing and improvement loops during the development of the report, here an example regarding the datasources. During testing the business pointed out that small differences in number of events were occuring between the new report on PowerBI and the datasource it should monitor. This was reported during a meeting of the project staff and the product owner of the \gls{ASTV} team. The error was reported by the product owner, and pointed out that the new Power BI report was still using the old data source.
While the data source remained largely the same, the data lineage changed slightly. Business and IT decided in late 2025 that he data should be consumed as a star schema to improve performance compared to the existing wide table. Consequently, the \gls{DQM} dashboard for message processing needed to follow this change in the data source. In order not to disturb the previous report, the data had to be gathered into a separate view in Athena. The separate view was also required in order to validate the new star schema \gls{infoblock}s rather than those of the previous wide table. Thus, separating the views was necessary to maintain clean, high-performing code, as adding the new query parts for the star schema to the wide table query would have slowed the query down and intertwined two different use cases.
